## 文章摘要

> 注：原文为 Twitter（X）帖子，无法直接抓取，本文主要依据 HN 讨论内容还原与总结。

一位在 Google 工作了 7 年的工程师（Justin Poehnelt）发帖称，自己因为创建并开源发布了一个 Google Workspace CLI（命令行工具）而遭到解雇。据其叙述，这个项目是在获得经理批准的情况下，发布在 Google 官方的 GitHub 组织里的，还获得了社区的好评，但他最终却被开除。被解雇的确切理由始终不甚清晰，但似乎与商标/品牌使用方面的顾虑有关。

围绕这件事，HN 上形成了针锋相对的两派观点。

**支持这位工程师的一方认为**：
- 这个工具发布在 Google 官方组织里，旁边还有 57 个类似的项目；
- 他的经理（Addy Osmani）曾公开宣传过这个工具，后来还主动从 Google 离职了；
- 该仓库至今仍然活跃，并且依然被链接在 Google 的官方开发者文档中；
- 这完全符合典型的开发者关系（DevRel）工作——创建展示 API 能力的开源示例。

**批评这位工程师的一方则认为**：
- 发布此类项目本应走完整的法务/组织审批流程；
- 未经明确授权使用 Google 品牌违反了公司政策；
- 时间点恰好与 Google 自己正在开发的官方 Workspace CLI 撞车；
- 可能造成了内部混乱，干扰了内部竞争项目的推进。

整件事折射出一个老问题：在大公司里，"按惯例做了对的事"和"踩了合规红线"之间，往往只隔着一条模糊且事后才被强调的界线。

## HN 评论精华

**cdata（约 721 赞，讨论主串）**：一位前 Googler 表示，在他任职期间（2015—2021 年），向 Google 管理的 GitHub 组织发布项目是标准做法。他质疑，相比一般的政策违规，直接解雇是否属于一种过度反应。

**qmarchi（高赞回复）**：提出相反视角，强调在 Cloud 产品线，任何使用"Google 名号"的东西都需要经过 IARC 委员会加法务的双重审批，暗示这位工程师"没有把事情的全部真相说出来"。

**whstl**：注意到那位经理已经离职，而且尽管所谓的违规存在，涉事仓库却并没有被删除——以此质疑 Google 对此事的真实在意程度到底有多高。

**gerdesj**：从企业文化角度评论，认为 Google 已经从当年那个颠覆式创新的初创公司，蜕变成一台"无趣的、只会赚钱的老牌印钞机"，开始惩罚那些敢于冒险的员工。
